# Kali â€” Internationalization (i18n)

Kali is built internationalizable from day one. The two shipped locales are
**English (en)** and **Spanish (es)**. Adding a third locale should be a
matter of dropping a JSON file in a folder â€” no code changes.

## Principles

1. **No hardcoded user-facing strings in code.** All UI text, consent
   reasons, tool descriptions, error messages, and settings labels are
   looked up via i18n keys.
2. **The core speaks keys, the frontend translates.** kali-core emits i18n
   keys (and any dynamic parameters) for any text that will be shown to the
   user. kali-web owns the locale catalogues and renders the final string.
   This keeps kali-core locale-agnostic and avoids duplicating translations
   across Python and TypeScript.
3. **Locale follows the OS by default.** The user can override in Settings,
   and the choice persists in `localStorage`.
4. **Markdown, code, and audio are not translated.** Only chrome (UI
   labels, prompts, buttons, settings) is translated. Agent responses are in
   whatever language the user wrote in.

## Frontend (kali-web)

- **Library:** `react-i18next` + `i18next` + `i18next-browser-languagedetector`.
- **Catalogues:** `src/locale/{en,es}/common.json`. Namespaces may be added
  later (`common.json`, `settings.json`, `consent.json`); for now one
  namespace is enough.
- **Language detection order:** `localStorage` (`kali.lang`) â†’ browser
  `navigator.language` â†’ fallback `en`.
- **Switching:** the Settings modal has a language selector that updates
  `i18n.changeLanguage` and persists to `localStorage`.

### Interpolation

`consent.reason.screenshot` above uses `{{reason}}` â€” the core passes the
reason as a parameter, the frontend interpolates. The reason itself is
another i18n key when possible (e.g.
`consent.reason.run_tests`), so the whole chain stays translatable.

## Core (kali-core)

The Python side does **not** translate. It emits structured payloads:

```python
# A consent request from kali-collar
{
  "event": "consent_request",
  "id": "req_abc",
  "tool": "run_command",
  "risk": "sensitive",
  "reason_key": "consent.reason.run_tests",   # i18n key
  "reason_params": {"command": "pytest"},     # interpolation
  "summary_key": "consent.summary.run_tests",
}
```

The frontend looks up `reason_key` in the active catalogue, interpolates
`reason_params`, and shows it in the ConsentModal.

Tool descriptions shown to the LLM are *not* i18n'd â€” the LLM gets the
canonical English description so it can reason consistently. Tool
descriptions shown in the UI (e.g. in a tooltip) are i18n keys.

## Consent and permission text

Consent text is the most sensitive area: the user must understand what they
are approving, in their language. Rules:

- Every `consent.reason.*` key has entries in `en` and `es`.
- If a tool needs a new reason, it adds a key to both catalogues in the same
  PR. CI should enforce that all locale files have the same keys (see
  below).
- The ConsentModal always shows all three buttons (`allow`, `no_capture`,
  `cancel`) translated, plus the localized reason.

## Voice

TTS and STT models are language-specific. Kali ships:
- **TTS:** `robot-es` (Spanish) by default. Other voices can be added by
  dropping a JSON + `.onnx`. The TTS language is independent of the UI
  language â€” you can have a Spanish UI with an English TTS voice.
- **STT:** `vosk-model-small-es-0.42` by default. Other Vosk models can be
  added. Same independence: UI language and STT language are separate config
  values.

## Dates and numbers

Use `Intl.DateTimeFormat` and `Intl.NumberFormat` with the active locale.
Do not format dates/numbers manually.

## CI checks

A lint script (`scripts/check-i18n.mjs`, Phase 1) enforces:

1. All catalogues (`en`, `es`) have the same set of keys.
2. No catalog has a key with an empty string.
3. No `t("â€¦")` call in the frontend references a key missing from `en`.

## Adding a new language

1. Copy `src/locale/en/common.json` to `src/locale/<lang>/common.json`.
2. Translate every value.
3. Add `<lang>` to the language selector in Settings.
4. Done â€” no other code changes.

## What is not translated

- Agent responses (the LLM answers in the user's input language).
- Code blocks and diffs shown in the canvas.
- Tool names in the UI (always English, for consistency with the LLM's tool
  definitions).
- Log messages (developer-facing, English).
- The project name "Kali".
